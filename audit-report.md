# Comprehensive Jekyll Web Template Code & Layout Audit Report

**Date:** July 12, 2026  
**Project:** Eric Kwon - Analytics Specialist Blog (`ericsoonyoulkwon.github.io`)  
**Target Architecture:** Jekyll (Minima Theme) / GitHub Pages  
**Auditor:** Gemini CLI  

---

## 1. Executive Summary

This audit report evaluates the codebase, structure, layouts, and style definitions of the Jekyll-based web template. The objective is to identify layout bugs, HTML/CSS specification violations, accessibility issues, browser compatibility errors, and structural problems that affect the site's deployment, SEO, performance, and user experience.

### Key Discoveries:
*   **Systemic Build Failure (Critical):** All blog posts in `_posts/` and the main homepage `index.md` completely lack Jekyll YAML front matter. Jekyll will copy these files as-is rather than compiling them into HTML. Consequently, `site.posts` is empty, causing the homepage and the `archive.md` page to display **zero posts**.
*   **Broken Social Sharing (Major):** The `_config.yml` lacks `url` and `baseurl` fields. Because of this, Jekyll's `absolute_url` filter fails. The social sharing buttons pass relative paths, making them completely broken on Facebook, Twitter, and LinkedIn.
*   **HTML Standard Violations (Major):** Style sheets and favicon link tags are embedded within `_layouts/post.html`, which is rendered inside the `<body>` element. This violates HTML standards and causes Flash of Unstyled Content (FOUC).
*   **Obsolete Features & Dead Code (Minor):** The sharing layout retains Google Plus (`gplus`), a platform shut down in 2019, leading to broken outgoing links. It also has dead CSS for Instagram and Pinterest that are not utilized in the HTML.
*   **Accessibility Copy-Paste Bugs (Minor):** Images in several recent posts contain completely irrelevant alt-text (`name-split-formula`) due to template copying. This severely harms WCAG compliance and search engine indexing.
*   **Browser Compatibility Hazards (Minor):** The custom cursor in `css/override.css` is declared using a raw, unencoded SVG with line breaks and unescaped spaces. Modern browser engines (Blink/WebKit) reject this format, causing the custom cursor to silently fail.

Below is the detailed breakdown of all audited items, categorized by severity, with exact remediation code blocks.

## 2. Audit Findings & Detailed Breakdown

### 2.1 Critical Severity Issues

#### 2.1.1 Lack of YAML Front Matter in Blog Posts and Homepage
*   **Location:** `index.md`, `_posts/*`
*   **Problem:** 
    A Jekyll-processed page or post **must** start with a YAML front matter block (enclosed between two triple-dash lines `---`). Every post in the `_posts/` directory and the main `index.md` starts with raw Markdown headers (e.g. `# Automating daily reports with VBA`) and contains no front matter at all.
*   **Impact:** 
    *   Jekyll will not compile the markdown files into HTML pages; it will copy them as static text/markdown.
    *   No layouts (like `post` or `home`) will be applied. The pages will render without the header, navigation links, layout wrappers, or footer.
    *   The posts will not be added to the `site.posts` collection. This means they will not appear in tags, RSS feeds, sitemaps, or on the archive page (`archive.md`), leaving the site completely empty.
*   **Remediation Plan:**
    Add proper YAML front matter to the top of `index.md` and all files in `_posts/`.
    
    *For `index.md` (Homepage):*
    ```markdown
    ---
    layout: home
    title: Home
    ---
    
    - Data Analytics and Data Science
    - Automation Innovator
    - Analytics Strategy
    - Data Governance & Quality
    - Continuous Improvement Leadership
    ```
    
    *For a Post (e.g., `_posts/2023-05-21-automating-daily-report-with-VBA.md`):*
    ```markdown
    ---
    layout: post
    title: "Automating daily reports with VBA"
    date: 2023-05-21 09:00:00 +0000
    categories: [MS Excel & VBA]
    tags: [VBA, Automation, Excel]
    ---
    
    Writing a macro to automate folder editing and Excel reports because I want to avoid doing it manually every day!
    
    ### So what is currently done manually?
    ...
    ```
    *(Note: You can remove the duplicate `# Title` and `#### Category: ...` from the body since the Jekyll layout automatically injects the title and category).*

### 2.2 Major Severity Issues

#### 2.2.1 Missing `url` and `baseurl` in `_config.yml`
*   **Location:** `_config.yml`
*   **Problem:**
    The `url` (e.g., `https://ericsoonyoulkwon.github.io`) and `baseurl` variables are not defined in `_config.yml`.
*   **Impact:**
    *   Any link or asset referencing `site.url` or using Jekyll's `absolute_url` filter (like `_includes/navlinks.html` and `_includes/sharelinks.html`) will fail to resolve.
    *   Canonical SEO tags generated by `jekyll-seo-tag` and RSS feeds generated by `jekyll-feed` will output invalid, relative-only links that prevent search engine indexing and RSS reader aggregation.
*   **Remediation Plan:**
    Open `_config.yml` and add the domain configurations:
    ```yaml
    title: "Eric Kwon - Analytics Specialist"
    author: "Eric Kwon"
    description: "Eric Kwon - Analytics Specialist"
    url: "https://ericsoonyoulkwon.github.io" # Add this line
    baseurl: ""                               # Add this line (empty if user page, or "/repo-name" if project page)
    ```

#### 2.2.2 Broken Social Sharing in `_includes/sharelinks.html`
*   **Location:** `_includes/sharelinks.html`
*   **Problem:**
    The social sharing URLs construct absolute links like so:
    `{% assign safeurl = site.url | append: page.url | url_encode %}`
    Since `site.url` is empty, `safeurl` evaluates to a relative path (e.g., `%2F2023%2F05%2F21%2Fautomating...html`).
*   **Impact:**
    When sharing on Facebook or LinkedIn, the sharing dialog opens with an invalid URL parameter that lacks a hostname. The APIs reject the relative URL, and sharing fails completely.
*   **Remediation Plan:**
    Ensure `site.url` is configured in `_config.yml`. In addition, rewrite the assignment to fall back to a safe default if `site.url` is missing:
    ```html
    {% assign site_url = site.url | default: "https://ericsoonyoulkwon.github.io" %}
    {% assign safeurl = site_url | append: page.url | url_encode %}
    ```

#### 2.2.3 HTML Standard Violations in `_layouts/post.html`
*   **Location:** `_layouts/post.html`
*   **Problem:**
    The layout inserts `<link href="/css/override.css" rel="stylesheet" type="text/css">` and `<link rel="shortcut icon" type="image/png" href="favicon.png">` directly in the body layout.
*   **Impact:**
    *   When Jekyll compiles a post, it wraps `post.html` inside the body of `default.html`. As a result, these links are rendered in the middle of the `<body>` tag.
    *   **Specification Violation:** Under W3C HTML specifications, `<link>` tags with relations other than stylesheet are not allowed in the body, and stylesheets belong in the `<head>`.
    *   **Broken Favicon Link:** The favicon link is hardcoded to `favicon.png`. When navigating nested blog post URLs, the browser resolves this relatively as `/2023/05/21/favicon.png` which produces a 404 error.
*   **Remediation Plan:**
    1.  Remove `<link>` elements from the body layout `_layouts/post.html`.
    2.  Add the `override.css` link globally to `_includes/head.html` inside the `<head>` tags.
    3.  Rely on the global favicon link already correctly configured in `_includes/head.html` (`<link rel="shortcut icon" type="image/png" href="/favicon.png">`).
    
    *Cleaned up top of `_layouts/post.html`:*
    ```html
    ---
    layout: default
    ---
    <article class="post h-entry" itemscope itemtype="http://schema.org/BlogPosting">
      <header class="post-header">
    ...
    ```

### 2.3 Medium Severity Issues

#### 2.3.1 Insecure Sharing and Twitter Username Bugs
*   **Location:** `_includes/sharelinks.html`
*   **Problems:**
    1.  The Facebook share link uses the insecure HTTP protocol (`http://www.facebook.com/share.php?u=...`).
    2.  The Twitter intent includes `&via={{ site.twitter_username }}`, but `twitter_username` is commented out in `_config.yml`.
*   **Impact:**
    1.  HTTP requests lead to unnecessary redirects to HTTPS and trigger modern browser security policies (mixed content warnings).
    2.  The sharing URL ends with a trailing, empty query parameter `&via=`, which looks unprofessional and is structurally invalid.
*   **Remediation Plan:**
    1.  Upgrade the Facebook sharing link to `https://`.
    2.  Conditionally output `&via=` only if the variable exists:
    ```html
    <!-- Facebook -->
    <div class="facebook" title="Share this on Facebook" onclick="window.open('https://www.facebook.com/share.php?u={{ safeurl }}','popup','width=600,height=600'); return false;"> ... </div>

    <!-- Twitter -->
    <div class="twitter" title="Share this on Twitter" onclick="window.open('https://twitter.com/intent/tweet?url={{ safeurl }}&text={{ safetitle }}{% if site.twitter_username %}&via={{ site.twitter_username }}{% endif %}','popup','width=600,height=600'); return false;"> ... </div>
    ```

#### 2.3.2 Non-Standard CSS Custom Cursor Encoding
*   **Location:** `css/override.css`
*   **Problem:**
    The custom cursor is declared in CSS as:
    ```css
    body {
      cursor: url("data:image/svg+xml;utf8,
        <svg xmlns='http://www.w3.org/2000/svg' width='32' height='32'>
          <polygon points='0,0 0,32 10,22 16,30 20,28 14,20 24,20'
            fill='white' stroke='black' stroke-width='2'/>
        </svg>") 0 0, auto;
    }
    ```
    This declaration contains raw line breaks, unescaped whitespace, and non-URL-safe characters (`<`, `>`, quotes) without URL-encoding.
*   **Impact:**
    Modern browser layout engines (Blink/WebKit in Chrome, Edge, and Safari) strictly enforce URI formatting for SVG cursors. Because this string contains unescaped special characters and linebreaks, the custom cursor fails to load completely, resulting in a silent console error or fallback to the OS cursor.
*   **Remediation Plan:**
    Use a fully URL-encoded (or Base64-encoded) string that is safe for all browsers, formatted onto a single line:
    ```css
    body {
      cursor: url("data:image/svg+xml;charset=utf8,%3Csvg%20xmlns%3D%27http%3A%2F%2Fwww.w3.org%2F2000%2Fsvg%27%20width%3D%2732%27%20height%3D%2732%27%3E%3Cpolygon%20points%3D%270%2C0%200%2C32%2010%2C22%2016%2C30%2020%2C28%2014%2C20%2024%2C20%27%20fill%3D%27white%27%20stroke%3D%27black%27%20stroke-width%3D%272%27%2F%3E%3C%2Fsvg%3E") 0 0, auto;
    }
    ```

### 2.4 Low Severity, Accessibility, & Usability Issues

#### 2.4.1 Image Alt-Text Copy-Paste Errors (Accessibility & SEO Bug)
*   **Location:** `_posts/2025-07-30-windows-startup-automation.md`, `_posts/2025-07-07-web-scraping-iherb-catalogue.md`
*   **Problem:**
    Several images in these newer posts have the alt-text hardcoded to `name-split-formula`. This was copied from an Excel string reconciliation post and never updated.
    *   In `windows-startup-automation.md`: Images showing shell startup folders, file shortcuts, and Chrome tabs all read "name-split-formula".
    *   In `web-scraping-iherb-catalogue.md`: The scraper output spreadsheets read "name-split-formula".
*   **Impact:**
    *   **Accessibility Violation (WCAG 2.1):** Screen-readers will announce "name-split-formula" for shell diagrams or scraping catalogs, creating a confusing and poor experience for visually impaired users.
    *   **SEO Deficit:** Google and other image search engines will incorrectly index these images, reducing search relevance.
*   **Remediation Plan:**
    Update the alt-text in each post to accurately reflect the image's content.
    
    *In `_posts/2025-07-30-windows-startup-automation.md`:*
    ```markdown
    L40:   ![Shell Startup Directory Screenshot](/images/shell-startup.png)
    L56:   ![Creating a Windows Shortcut Screenshot](/images/create-shortcut.png)
    L72:   ![Windows Startup Folder Directory](/images/startup-folder.png)
    L132:  ![Chrome Multiple Tabs Properties](/images/chrome-tabs-shortcut.png)
    ```

    *In `_posts/2025-07-07-web-scraping-iherb-catalogue.md`:*
    ```markdown
    L615:  ![Scraped iHerb Product Catalogue Excel Output](/images/iHerb_cat.png)
    L620:  ![Naver Shopping Price Comparison Sheet](/images/car_enriched_naver.png)
    ```

#### 2.4.2 Obsolete Google Plus Sharing Button
*   **Location:** `_includes/sharelinks.html`
*   **Problem:**
    Includes a button to share posts to Google Plus (`gplus`). Google Plus was shut down in April 2019.
*   **Impact:**
    The button is useless, and clicking it takes the user to a dead link or standard 404 page, degrading the site's perceived quality.
*   **Remediation Plan:**
    Remove the `gplus` `<div>` block from the HTML. You can also clean up the CSS rule `#share-buttons > div.gplus:hover > svg {fill: #dd4b39;}` and related sizing rules from the style tag.

#### 2.4.3 Dead CSS in Sharing Styles
*   **Location:** `_includes/sharelinks.html`
*   **Problem:**
    The inline stylesheet defines classes for `instagram` and `pinterest`, but no such sharing buttons are configured in the HTML markup.
*   **Impact:**
    Unused CSS styles increase the page weight unnecessarily, although the overhead is minimal.
*   **Remediation Plan:**
    Remove the unused classes from the stylesheet:
    ```css
    /* Delete these selectors if Instagram and Pinterest sharing are not planned */
    #share-buttons > div.instagram:hover > svg {fill: #C73B92;}
    #share-buttons > div.pinterest > svg {height: 20px; margin-top: 9px;}
    ```

#### 2.4.4 Custom Cursor UX and Inconsistent Delivery
*   **Location:** `css/override.css`
*   **Problem:**
    Forcing a custom cursor globally on a standard text-based blog degrades usability. Additionally, since `override.css` is only loaded in the post layout, the custom cursor will suddenly disappear when users navigate back to the home or archive pages.
*   **Impact:**
    This results in a jarring, inconsistent cursor animation on different pages and interferes with OS cursor accessibility features.
*   **Remediation Plan:**
    It is highly recommended to **remove** the custom cursor rule entirely to align with best practices in UX and accessibility. If a custom cursor is absolutely desired, include `override.css` in `_includes/head.html` so it applies consistently across all pages.

#### 2.4.5 Redundant CSS Highlighting Files
*   **Location:** `_includes/head.html`
*   **Problem:**
    The head template imports two separate highlight.js CSS stylesheets:
    ```html
    <link rel="stylesheet" href="/js/highlightjs/styles/github.css">
    <link rel="stylesheet" href="/js/highlightjs/styles/ssms.css" />
    ```
*   **Impact:**
    *   Loading two different themes will cause the styles from `ssms.css` to overwrite those in `github.css`.
    *   This is redundant and increases page load times.
*   **Remediation Plan:**
    Choose one theme and delete the other. If you prefer the SQL Server style, keep only `ssms.css`:
    ```html
    <!-- Only load the single preferred highlighting theme -->
    <link rel="stylesheet" href="{{ "/js/highlightjs/styles/ssms.css" | relative_url }}" />
    ```

#### 2.4.6 Duplicate Fallbacks in `_includes/navlinks.html`
*   **Location:** `_includes/navlinks.html`
*   **Problem:**
    If a blog post has no previous post (oldest) or no next post (newest), the navigation links default to displaying a "Blog Archive" fallback on that side.
*   **Impact:**
    If the blog only contains a single post, both the left (previous) and right (next) navigation blocks will display the exact same fallback "Blog Archive" link, which is repetitive and poorly designed.
*   **Remediation Plan:**
    Make the layout cleaner by completely hiding the nav section or conditionally outputting the fallback only on one side (or disabling the element rather than repeating the link):
    ```html
    <hr>
    <div class="post_navi">
      {%- if page.previous.url -%}
      <a class="post_navi-item nav_prev" href="{{ page.previous.url }}" title="{{ page.previous.title }}">
        <div class="post_navi-arrow">&lt;</div><div class="post_navi-label">Previous Post</div><div><span>{{ page.previous.title }}</span></div>
      </a>
      {%- else -%}
      <a class="post_navi-item nav_prev" href="{{ '/archive/' | relative_url }}" title="Blog Archive">
        <div class="post_navi-arrow">&lt;</div><div class="post_navi-label">Blog Archive</div><div><span>All posts archive</span></div>
      </a>
      {%- endif -%}
      
      {%- if page.next.url -%}
      <a class="post_navi-item nav_next" href="{{ page.next.url }}" title="{{ page.next.title }}">
        <div class="post_navi-arrow">&gt;</div><div class="post_navi-label">Next Post</div><div><span>{{ page.next.title }}</span></div>
      </a>
      {%- endif -%}
    </div>
    ```

#### 2.4.7 Hardcoded Domain URLs in Posts
*   **Location:** `_posts/2023-06-02-selecting-data-transacted-on-previous-business-day.md` (L73), `_posts/2023-05-21-automating-daily-report-with-VBA.md` (L24)
*   **Problem:**
    Links between posts are hardcoded to the production URL (`https://ericsoonyoulkwon.github.io/2023/...html`).
*   **Impact:**
    *   Prevents testing links in a local development environment (clicking the link redirects the developer from `localhost:4000` to the production site).
    *   Breaks portability if the repository is renamed or moved to a custom domain.
*   **Remediation Plan:**
    Use Jekyll relative URL tags to reference internal resources:
    ```markdown
    [automating the daily task]({{ "/2023/05/21/automating-daily-report-with-VBA.html" | relative_url }})
    ```

#### 2.4.8 Tag-Grouped Archive Limitations
*   **Location:** `archive.md`
*   **Problem:**
    The archive layout is structured by tags:
    ```html
    {% for tag in site.tags %}
      <h3>{{ tag[0] }}</h3>
      ...
    {% endfor %}
    ```
    In `_config.yml`, every post has a default tag of `Other`.
*   **Impact:**
    Unless tags are explicitly declared in each post's front matter, every blog post will be grouped under a single heading `Other`. This does not provide a standard chronological index, which is highly recommended for blogs.
*   **Remediation Plan:**
    Add a true chronological archive to `archive.md` (or combine chronological listing with tag lists):
    ```markdown
    ---
    layout: page
    title: Blog Archive
    ---
    
    ### Chronological Archive
    <ul>
      {% for post in site.posts %}
        <li><a href="{{ post.url | relative_url }}">{{ post.date | date: "%B %d, %Y" }} - {{ post.title }}</a></li>
      {% endfor %}
    </ul>
    ```

## 3. Summary of Recommended Actions

1.  **Run a Batch Script or Edit Post Files:** Add a standardized YAML front matter block to the top of all markdown files in `_posts/` and to `index.md`.
2.  **Update Config:** Add `url` and `baseurl` definitions to `_config.yml`.
3.  **Refactor Headers:** Move `/css/override.css` link into `_includes/head.html` and remove the local `<link>` tags from `_layouts/post.html`.
4.  **Polish Sharing Codes:** Remove the Google Plus container, correct the Facebook URL protocol, and use fallback Liquid logic for absolute sharing URLs.
5.  **Clean up Accessibility:** Correct the `alt-text` attributes for images in `windows-startup-automation.md` and `web-scraping-iherb-catalogue.md`.
6.  **Fix Custom Cursor SVG:** Percentage-encode the custom cursor SVG in `css/override.css`.

Implementing these steps will bring the site to full standards compliance, resolve build omissions, optimize page loads, restore social media integrations, and correct critical SEO and accessibility bugs.
